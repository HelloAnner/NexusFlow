# 收件箱 / 待办（Inbox）

> 参考 Linear Inbox 和 Slack Activity，将所有需要用户响应的事项集中到一个地方处理，支持批量操作和键盘快捷处理。

## 1. 页面目标

作为系统的消息与待办中心，让用户可以批量、快速、不间断地处理所有需要响应的事项。

## 2. 整体布局

```
┌────────────────────────────────────────────────────────────────┐
│  Inbox                  [ 全部已读 ] [ 批量处理 ] [ 设置 ]      │
├────────────────────────────────────────────────────────────────┤
│  [ 待办 ] [ 通知 ] [ 已处理 ]                                   │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌────────────────────────────────────┐  ┌─────────────────┐  │
│  │                                    │  │                 │  │
│  │  待办/通知列表                      │  │  详情预览面板   │  │
│  │                                    │  │  （可选展开）   │  │
│  │                                    │  │                 │  │
│  └────────────────────────────────────┘  └─────────────────┘  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 3. Tab 分类

### 3.1 待办（Action Required）

需要当前用户执行动作的事项：

- 待确认任务
- 待审批跨部门协调
- 待填报进度
- 待提交成果
- 待验收任务
- 待处理冲突
- 待审核注册（SA）

### 3.2 通知（Notifications）

只需知晓的事项：

- 被分配为负责人
- 被加入任务
- 任务状态变更
- 成果被确认/退回
- 任务被归档

### 3.3 已处理（History）

历史已处理待办和已读通知，支持按时间筛选。

## 4. 列表项设计

> 参考 Linear Inbox 的紧凑列表项。

每行包含：

- 未读指示器（左侧蓝色竖线）。
- 类型图标（审批/填报/验收/冲突/通知）。
- 标题：清晰描述需要做什么。
- 来源对象：关联任务/项目名称。
- 时间：多久前。
- 风险等级（高/中/低）。
- 快捷操作按钮（根据类型动态）：同意 / 拒绝 / 查看 / 稍后提醒。

## 5. 批量操作

- 支持多选（checkbox 或 Shift + 点击）。
- 批量操作栏：
  - 批量同意
  - 批量拒绝
  - 批量标记已读
  - 批量忽略
  - 批量稍后提醒

## 6. 键盘操作

| 快捷键 | 作用 |
|---|---|
| `J / K` | 上下移动 |
| `Enter` | 打开详情或执行主操作 |
| `A` | 同意 |
| `R` | 拒绝 |
| `S` | 稍后提醒 |
| `X` | 选择/取消选择 |
| `Shift + A` | 全选 |

## 7. 详情预览面板

- 点击待办后，右侧展开预览面板。
- 面板展示关联对象的摘要信息和处理操作。
- 对于简单审批，可直接在面板内完成同意/拒绝。
- 对于复杂任务，提供“打开完整详情”按钮。

## 8. 通知偏好设置

- 用户可配置接收方式：站内 / 邮件 / 短信。
- 可配置哪些事件产生通知。
- 可配置是否显示在 Home 组件。

## 9. 移动端适配

- 全屏列表，顶部 Tab 切换。
- 点击待办后进入处理页。
- 左滑快捷操作：已读 / 稍后 / 忽略。

## 10. 待办处理流程

- 用户进入 Inbox 后默认定位到第一条未处理待办，列表项获得键盘焦点，右侧预览面板同步展示详情。
- 简单事项（确认、已读、稍后提醒）允许在列表行内直接处理；复杂事项（审批、冲突、验收退回）必须在详情面板中展示上下文后处理。
- 处理成功后，当前项从待办列表移入已处理，焦点自动移动到下一条；如果列表为空，展示完成态。
- 处理失败时保持当前项选中，展示失败原因，并允许重试或打开完整详情。
- “稍后提醒”需要选择时间，默认提供 1 小时后、明天上午、本周五三个快捷项。
- 批量拒绝、批量忽略必须填写原因，原因写入对应对象的 Activity。

## 11. 信息与去重

- 同一个任务产生多个通知时，待办 Tab 只保留需要动作的最高优先级事项，其他只读通知归入 Notifications。
- 同一对象的连续状态变化在 Notifications 中可折叠为一组，例如“任务 A 有 3 条更新”。
- 列表标题必须以动词开头，直接告诉用户要做什么，例如“确认任务发布时间”而不是“任务状态变更”。
- 来源对象展示层级为“项目 / 任务 / 分工”，点击任一层级可打开对应详情。
- 时间展示使用相对时间，超过 7 天显示具体日期。

## 12. 权限与边界

- 用户只能处理分配给自己的待办；管理员可代处理时必须显示“代处理”标记并要求填写原因。
- 已处理 Tab 默认只展示最近 30 天，避免历史数据拖慢列表；更久历史通过筛选加载。
- 通知偏好只影响未来通知，不删除已产生通知。
- 如果关联对象已被删除或用户失去权限，列表项仍保留简要记录，但操作按钮置灰并说明原因。
- 离线或 WebSocket 断开时顶部显示同步状态，用户仍可查看已加载待办，但处理动作需要后端确认后才移出列表。

## 13. 当前落地切片

本阶段先把 `/notifications` 从基础表格升级为 Inbox 的通知处理视图，不新增后端通知偏好模型，先复用现有 `GET /api/notifications` 和 `POST /api/notifications/{id}/read` 完成可验证闭环。验收点如下：

- 页面必须提供“全部 / 未读 / 已读 / 权限受限”筛选，未读筛选只展示 `read_at` 为空的通知，权限受限筛选只展示 `payload.redacted=true` 或无可打开入口的通知。
- 通知列表必须支持多选；选中后显示批量操作栏，展示已选数量、未读数量，并支持批量标记已读。
- 批量标记已读必须顺序调用 `POST /api/notifications/{id}/read`；如果部分失败，保留当前选择并展示失败数量或后端错误。
- 点击列表项必须在右侧详情预览面板展示通知类型、状态、来源入口、脱敏原因、原始 payload 摘要和处理动作；复杂对象保留“打开完整详情”入口。
- 单条与批量已读成功后刷新列表，当前选择中已处理项自动移除；如果当前详情项被处理，详情面板更新为已读状态。
- 设置入口先提供本地通知偏好面板，支持站内 / 邮件 / 短信、是否在首页显示、关注事件类型，并保存到 `localStorage`；面板必须明确“只影响本机偏好预设，后端偏好模型待接入”。

## 14. 当前落地切片：待办处理队列

本阶段继续把 `/todos` 从基础左右栏升级为 Inbox 的待办处理队列，复用现有 `GET /api/todos` 和 `POST /api/todos/{id}/complete`，并通过 `todo_items.payload` 记录完成原因，不新增表结构。验收点如下：

- 页面必须提供待处理、已完成、全部、逾期、今日到期筛选；逾期和今日到期从 `due_at` 前端计算，不能影响后端列表兼容性。
- 待办列表必须支持多选；选中后显示批量操作栏，展示已选数量、待处理数量、逾期数量，并支持批量完成。
- 单条或批量完成必须要求填写完成原因；前端对每条待办调用 `POST /api/todos/{id}/complete` 并传入 `completion_reason`，后端将原因写入 `payload.completion_reason`、`payload.completed_at` 和 `payload.completed_by`。
- 批量完成如果部分失败，必须保留当前选择和原因，展示失败数量或后端错误，不能静默吞掉。
- 右侧详情预览必须展示待办类型、状态、截止时间、目标对象、完成原因、原始 payload 摘要和处理入口；已完成待办以只读方式展示完成信息。
- 完成成功后刷新列表；当前筛选为待处理/逾期/今日到期时，已完成项从列表移出，空列表展示完成态。

## 建议组件

Inbox 是用户处理待办与通知的“流水线”，交互密度高，必须保证列表响应迅速、操作触手可及。建议整体以 **shadcn/ui** 作为基础控件，并通过 Tailwind/CSS 变量强制映射到 NexusFlow 色板：列表行背景 `--bg-secondary`（#FFFFFF），hover 态 `--bg-tertiary`（#F5F4F2），未读指示使用 `--info`（#3B82F6）竖线，风险 badge 按语义色 `--error`/`--warning`/`--success` 区分高/中/低，文字层级使用 `--text-primary`（#1A1A1A）与 `--text-muted`（#7A7A7A）。禁止为不同待办类型引入额外彩色图标背景，统一使用单色线图标或语义色小点。

顶部 Tab 切换使用 shadcn/ui `Tabs`，包含“待办 / 通知 / 已处理”，激活态下划线颜色设为 `--text-primary`，保持极简。操作按钮组（全部已读、批量处理、设置）使用 shadcn/ui `Button` 的 `ghost` 与 `outline` 变体，主按钮（批量处理）填充 `--text-primary`。筛选与排序使用 `DropdownMenu`，下拉背景 `--bg-secondary`，选项 hover `--bg-tertiary`，分割线 `--border-subtle`。

列表项建议自研 **Inbox List Item 组件**，固定行高 56px，与全局列表行高规范一致。每行左侧 3px 未读竖线（未读时显示 `--info`），已读无竖线。类型图标使用 Lucide 线图标并统一为 `--text-muted`，包括审批、填报、验收、冲突、通知等类型。标题使用 `--text-primary` 并加粗未读项，来源对象与时间使用 `--text-muted`。风险等级 badge 使用语义色：`--error` 高、`--warning` 中、`--success` 低。hover 行背景过渡为 `--bg-tertiary`，过渡时长 150ms；同时右侧渐显快捷操作按钮（同意/拒绝/查看/稍后提醒），按钮使用 shadcn/ui `Button` 的 `ghost` 变体，尺寸 `sm`。复杂列表使用 **TanStack Virtual** 虚拟滚动，目标首屏 50 项内渲染完成。

批量操作通过 shadcn/ui `Checkbox` 实现多选，Shift + 点击连续多选通过自定义事件处理，选中行背景 `--bg-tertiary`。批量操作栏悬浮在列表上方，背景 `--bg-secondary`、底部阴影 `shadow-sm`、边框 `--border-subtle`，操作按钮包括批量同意、批量拒绝、批量标记已读、批量忽略、批量稍后提醒，危险操作使用 `--error` 文字色而非填充色，避免大面积红色造成焦虑。

键盘操作使用 **react-hotkeys-hook** 注册 `J/K`（上下移动）、`Enter`（打开详情或执行主操作）、`A`（同意）、`R`（拒绝）、`S`（稍后提醒）、`X`（选择/取消选择）、`Shift + A`（全选）。命令面板场景可复用 **cmdk** 做动作搜索，但 Inbox 内部导航优先使用自定义 hook，避免与全局命令面板快捷键冲突。快捷键需在可编辑元素上禁用，防止与输入框冲突。移动端左滑快捷操作使用自定义 touch 事件 + CSS transform，滑动阈值 80px，操作按钮背景 `--bg-tertiary`，图标颜色按操作语义：`--success`（已读）、`--warning`（稍后）、`--text-muted`（忽略）。

详情预览面板桌面端使用 shadcn/ui `Sheet`（右侧滑出，宽度 480px），移动端使用 **vaul** 全屏抽屉。面板背景 `--bg-secondary`，内部分割线 `--border-subtle`。面板展示关联对象摘要信息和处理操作，对于简单审批可直接在面板内完成同意/拒绝，复杂任务提供“打开完整详情”按钮。审批类操作使用 shadcn/ui `AlertDialog` 做二次确认，结果反馈使用 **Sonner**，成功提示使用 `--success`，失败使用 `--error`，保持语义色一致。通知偏好设置使用 shadcn/ui `Dialog` 或 `Sheet`，包含接收方式（站内/邮件/短信）`Switch`、事件类型 `Checkbox`、首页组件显示 `Switch`。

未读数与 Inbox 状态建议通过 WebSocket 或轮询实时同步，顶部铃铛徽章与侧边栏 Inbox 入口保持一致。已处理 Tab 支持按时间筛选与搜索。

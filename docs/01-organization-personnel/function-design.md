# 组织与人员功能设计

## 1. 功能目标

组织与人员模块用于支撑任务派发、权限判断、人员负载计算和管理统计。它不是单纯的人事档案，而是工作协同系统的基础数据中心。

模块目标：

- 建立中心内人员、组织、项目、技能、角色的统一档案。
- 支持任务派发时快速定位合适人员。
- 支持根据组织和项目归属自动计算数据范围。
- 支持人员状态和工时配置参与负载计算。
- 支持人员历史归属追溯，保证历史任务不因组织调整而失真。
- 支持邀请注册后自动创建人员档案并绑定组织、角色和项目。
- 支持 SA 超级管理员维护人员全部信息。

## 2. 组织管理

### 2.1 组织层级

组织层级包括：

- 公司。
- 二级单位。
- 三级单位/中心。
- 部门。
- 创新工作室。

主组织必须形成树形结构。创新工作室可作为横向组织挂在中心或部门下，但人员归属支持多选。

### 2.2 组织字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 组织名称 | 文本 | 是 | 页面展示名称 |
| 组织编码 | 文本 | 是 | 系统唯一编码 |
| 组织类型 | 枚举 | 是 | 公司、二级单位、中心、部门、工作室 |
| 上级组织 | 组织引用 | 否 | 顶级组织为空 |
| 负责人 | 人员 | 否 | 用于默认审批和统计 |
| 副负责人 | 人员多选 | 否 | 可参与审批 |
| 技术监督 | 人员多选 | 否 | 可参与技术类任务协调 |
| 默认审批人 | 人员多选 | 否 | 跨部门协调默认候选 |
| 是否启用 | 布尔 | 是 | 停用后不可新增归属 |

### 2.3 组织变更

组织变更要求：

- 组织名称修改应实时影响展示。
- 组织层级变更只影响新任务的组织口径。
- 历史任务保留创建时组织快照。
- 组织停用后，历史数据仍可查询。
- 删除组织前必须检查是否存在人员、任务、资料、配置引用。

## 3. 人员管理

人员来源：

- SA 超级管理员直接创建。
- 授权管理员在管理范围内创建。
- 用户通过邀请链接自助注册后自动创建。
- 已有账号通过邀请链接补充组织、角色或项目归属。

### 3.1 人员字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 姓名 | 文本 | 是 | 真实姓名 |
| 工号/账号 | 文本 | 是 | 唯一标识 |
| 登录账号 | 文本 | 是 | 系统登录映射 |
| 主组织 | 组织 | 是 | 决定默认管理归属 |
| 创新工作室 | 组织多选 | 否 | 横向组织归属 |
| 系统角色 | 角色多选 | 是 | 决定操作权限 |
| 人员等级 | 枚举 | 是 | 管理等级或专业等级 |
| 项目归属 | 项目多选 | 否 | 用于项目视图和权限 |
| 技能标签 | 标签多选 | 否 | 用于派发匹配 |
| 工作状态 | 枚举 | 是 | 在岗、出差、培训、休假、离职 |
| 每日标准工时 | 数字 | 是 | 默认 8 小时 |
| 是否可派发 | 布尔 | 是 | 离职或长期不可用时关闭 |
| 账号状态 | 枚举 | 是 | 待审核、启用、禁用、锁定 |
| 邀请来源 | 邀请链接 | 否 | 通过邀请注册时记录 |
| 默认首页角色 | 角色 | 否 | 多角色时的默认入口 |

### 3.2 人员等级

管理等级：

- 中心主任/书记。
- 中心副主任/技术总监。
- 部门主任/技术监督。
- 部门副主任。
- 项目负责人。
- 普通员工。

专业等级：

- 助理工程师。
- 三级工程师。
- 二级工程师。
- 一级工程师。

管理等级和专业等级可并存。系统角色决定权限，人员等级主要用于展示、筛选和统计。

### 3.3 技能标签

技能标签按分组维护：

- 研发类：前端、后端、Web 端、手机端、PC 端、运维、Python、Java、Go、SQL。
- 文职类：PPT 制作、报告撰写、论文、专利、申报材料、党建材料。
- 需求对接类：技术支持、系统培训、需求收集、运维客服。

标签要求：

- 支持启用和停用。
- 支持同义词备注，例如 Web 端和前端。
- 支持在人员列表、人员选择器、任务派发中筛选。
- 支持按标签统计人员数量。

## 4. 人员项目归属

项目归属用于说明人员当前参与哪些项目，不等同于任务成员。

项目归属字段：

- 项目名称。
- 项目身份：负责人、核心成员、参与成员、支持人员。
- 负责工作。
- 加入时间。
- 退出时间。
- 是否当前有效。

项目归属影响：

- 项目负责人查看项目成员。
- 项目任务派发候选人推荐。
- 项目维度甘特图和报表。

## 5. 人员状态

人员状态会影响任务派发。

| 状态 | 是否可派发 | 负载处理 |
| --- | --- | --- |
| 在岗 | 是 | 正常计算 |
| 出差 | 视任务类型 | 出差期间可接出差相关任务，其他任务提示 |
| 培训 | 视配置 | 默认不推荐派发新工作 |
| 休假 | 否 | 与任务重叠时提示冲突 |
| 离职 | 否 | 不可新派发 |

状态可以由任务自动驱动，例如创建休假任务后，该人员在对应日期显示休假状态。

## 6. 关键业务规则

- 人员必须属于一个主组织。
- 一个人可以属于多个工作室和多个项目。
- 邀请注册创建的人员必须自动绑定邀请模板中的主组织和默认角色。
- 邀请模板配置项目时，注册后自动写入项目归属、项目身份和负责工作。
- 待审核人员不可访问业务数据，只能查看资料完善和审核状态。
- 任务派发时必须校验人员是否可派发。
- 人员离职后保留历史任务和资料署名。
- 组织、角色、项目归属变更需要记录变更人、变更时间和变更前后值。
- 管理者只能维护自己权限范围内的人员信息。
- SA 超级管理员可维护人员全部字段，但修改角色、组织、隐藏授权和账号状态必须填写原因并写入审计。

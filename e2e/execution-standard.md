# E2E 执行与验收规范

## 结果状态

| 状态 | 含义 |
| --- | --- |
| `PASS` | 页面截图和 API 断言全部通过。 |
| `FAIL_UI` | API 正常但页面交互、展示、截图不符合预期。 |
| `FAIL_API` | 页面可操作但 API 状态码、结构或后置数据错误。 |
| `FAIL_PERMISSION` | 权限、数据范围、隐藏项目或待审核隔离不符合预期。 |
| `BLOCKED_BY_IMPLEMENTATION` | PRD 要求存在，但当前页面或接口未实现。 |
| `BLOCKED_BY_DATA` | 缺少必要测试账号、组织、项目、任务或资料。 |

## 每条用例的最小证据

- 登录态截图。
- 进入功能前截图。
- 操作中关键状态截图，例如弹窗、确认框、表单错误、冲突检查。
- 操作完成后截图。
- API 请求、响应状态码和关键字段记录。
- 如产生数据变更，必须后置查询确认。

## Agent Browser 基础命令

```bash
agent-browser open http://127.0.0.1:8089/login
agent-browser snapshot -i
agent-browser fill @账号输入框 "Anner"
agent-browser fill @密码输入框 "1"
agent-browser click @登录按钮
agent-browser wait --url "**/"
agent-browser screenshot e2e/artifacts/{run-id}/{case-id}/01-login.png
```

实际执行时必须使用 `snapshot -i` 获取当前页面 refs，不能假设 refs 固定。

## API 验证规则

登录后保存 token：

```bash
TOKEN="$(curl -s http://127.0.0.1:8089/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"login_name":"Anner","password":"1"}' | jq -r '.token')"
```

API 验证必须检查：

- HTTP 状态码。
- JSON 是否可解析。
- 列表接口是否有 `items`。
- 详情接口是否有稳定 `id`。
- 写接口后是否能通过查询读回。
- 权限拒绝是否是 401/403，而不是空白页或 500。

## 截图判定规则

截图不是装饰，必须逐一确认：

- 页面标题与当前功能一致。
- 关键字段、状态标签、按钮、筛选项存在。
- 表单错误或权限提示文本清晰。
- 数据变更前后有可见差异。
- 空态、加载态、禁用态、异常态分别截图。
- 移动端用例需设置移动视口后截图。

## 角色集

全量测试需要至少准备这些账号或通过邀请注册流程创建：

- SA 超级管理员。
- 系统管理员。
- 中心主任/书记。
- 中心副主任/技术总监。
- 部门主任/技术监督。
- 部门副主任。
- 项目负责人。
- 任务负责人。
- 普通员工。
- 待审核用户。
- 隐藏项目未授权用户。

## 测试数据集

全量测试至少需要：

- 两级以上组织树，含中心、部门、工作室。
- 两个以上项目，其中一个隐藏项目。
- 每类任务：科研、报告、协同支持、出差、休假、个人工作记录、后补任务。
- 至少一个跨部门任务。
- 至少一个待协调审批单。
- 至少一个工时超载冲突、全天冲突、休假冲突。
- 至少一个必需资料缺失任务和一个资料完整任务。
- 至少一个待审核注册申请。
- 至少一组配置草稿、发布版本和停用版本。
